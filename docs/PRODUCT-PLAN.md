## 0. 执行摘要

### 0.1 产品不是家庭产品起步

首阶段面向**成年人个人用户**，覆盖独身者与有家庭者。产品必须在用户不邀请任何家人的情况下独立成立：

```text
个人先长期自用
→ 感受到它足够聪明、可靠、有记忆、能推进事情
→ 建立对能力、隐私和安全的信任
→ 可选：推荐或邀请自己关心的老人和孩子
→ 可选：形成经授权的家庭/亲友协作网络
```

老人和孩子是后续重点适配对象，不是首阶段的产品成立条件。家庭身份不是获得完整能力的前提。

### 0.2 产品核心

```text
微信客服低摩擦触达＋小程序私密深度空间
＋长期理解一个人
＋保留阶段变化和事实来源
＋给出不迎合、可执行的现实方案
＋把方案转成行动并继续跟进
＋用户能查看、纠正、删除和控制记忆
```

### 0.3 首版必须证明的事情

首版不是记录工具，也不是后台工程。必须真实证明：

1. 用户可以通过微信小程序发送文字或语音并获得流式AI回复；
2. 用户明确陈述的个人信息形成可查看、可删除的记忆；
3. 第二轮或后续对话真实检索并使用相关记忆，回答因此发生正确变化；
4. 新信息与旧信息冲突时保留时间线，不机械复述旧结论；
5. 一条建议可转成行动，并在之后收到跟进；
6. 全链路身份、数据隔离、日志脱敏和删除可验证。
7. 用户始终看得见当前入口与隐私模式；人工访问只能由用户对指定内容临时授权并留下审计。

只保存消息、生成摘要、建立数据表或做聊天UI，都不算完成AI闭环。

### 0.4 技术结论

- 入口：微信客服作为门口、轻对话和提醒通道，原生微信小程序作为深度对话、长期记忆和隐私控制空间；
- 接续：两端经服务端可信身份绑定共享会话与记忆，使用短期单次跳转凭证续接上下文，链接中不携带聊天正文；
- 应用底座：CloudBase身份、云函数、数据库、对象存储和日志；
- 模型接入：CloudBase作为首版模型网关，同时支持内置与自定义模型；
- 产品智能：记忆治理、ContextBuilder、模型路由、策略、Skill、工具调用和行动闭环由自有领域层掌握；
- 智能体内核：吸收Hermes的Skill/记忆/纠错闭环和Grok Build的Agent Loop/工具调度/审批思想，实现受控的`Hermes-lite`，不把整套个人本机Runtime搬入多租户后端；
- 服务端：小程序只能调用自有Conversation API/云函数，不得绕过策略层直接调用模型；
- 后台：首版只做支撑AI闭环的最小运营能力，完整React后台后置；
- 人工访问：默认没有查看原始对话的产品入口；用户主动提交指定片段排障后，才创建有范围、有期限、可撤销和可审计的临时访问授权；
- 隐私：真实用户内容、画像、标识、媒体、日志、密钥和导出文件不得进入Git。

### 0.5 v2.0被替代的路线

以下旧路线停止执行：

- “首版不接LLM”；
- 先完成完整后台，再做AI；
- 先做记录CRUD和家庭分享，再做个人分身；
- 把家庭中坚力量仅视为购买者或管理员；
- 把语音、图片、提醒本身当作核心差异。

旧POC中可复用的输入、CloudBase、领域测试和权限代码保留，但阶段顺序以v3.2为准。

## 1. 产品定位与目标用户

### 1.1 一句话定位

**一个在微信里长期了解你、给出贴合现实的方案，并陪你把事情做成的个人AI助理。**

### 1.2 首批用户

- 中国大陆成年人个人用户；
- 独身、已婚、有无孩子均可；
- 愿意持续使用AI处理生活、工作、关系、学习或个人成长问题；
- 不要求会配置模型、Prompt、知识库或自动化工具；
- 对“每次重新解释背景”和“建议正确但不适合自己”有明显痛点。

### 1.3 后续用户

#### 老人

- 语音优先、大字号、少层级、明确反馈；
- 重点是理解、陪伴和现实协助，不是监控；
- 老人拥有自己的账户和数据自主权；
- 家属邀请、付款或关心不等于默认读取全部对话。

#### 孩子

- 按年龄适配表达、能力和安全边界；
- 重点支持好奇心、表达、创造和成长，不替代家长与老师；
- 必须有监护同意、内容安全、使用时长和高风险升级机制；
- 不能以“监护”为理由默认向家长展示全部原始对话。

### 1.4 首阶段核心使用场景

首版不做大而全功能菜单，而围绕一个持续循环：

```text
用户用一句话或一段语音说出当前问题
→ AI结合个人历史理解真实约束
→ 给出结论、理由和可执行下一步
→ 用户选择是否形成行动
→ 系统在约定时间跟进结果
→ 下一次建议使用这次结果继续改进
```

场景可以来自工作、生活、关系、学习或个人成长，但产品体验保持同一个闭环。

### 1.5 不做的定位

- 不宣传自研基础大模型全面超过豆包、元宝、Kimi或千问；
- 不做纯情绪迎合型AI伴侣；
- 不做医疗诊断、处方、心理治疗或紧急救援替代品；
- 不做家庭监控、儿童监控或老人定位工具起步；
- 不以公开社区、广告和内容流作为首版增长方式；
- 不要求用户先批量导入全部人生资料才能使用。

## 2. 核心竞争力与成功指标

### 2.1 四层壁垒

#### 理解层

- 区分稳定事实、临时状态、人生阶段、偏好、关系边界和决策规则；
- 记住与当前问题相关的信息，不把全部画像塞进Prompt；
- 新旧信息冲突时保留时间线和来源；
- 只从用户原始输入学习，不把AI回复反写成用户事实。

#### 方案层

- 先给结论，再给关键依据和备选路径；
- 明确区分事实、推断和假设；
- 方案结合用户现实约束、历史选择和结果反馈；
- 不迎合，不用鸡汤替代判断。

#### 行动层

- 建议可以转成行动项；
- 行动有负责人、时间、状态和结果；
- 系统按用户确认的时间跟进；
- 结果进入下一轮方案，但不能自动改写成稳定画像。

#### 信任层

- 用户能查看“记住了什么、来自哪里、何时更新”；
- 用户能修改、删除和暂停记忆；
- 临时对话不读也不写长期记忆；
- 分享必须明确、可预览、可撤回、最小范围。

### 2.2 北极星指标

**连续使用后，用户认为“它给我的方案明显比第一次更贴合，而且我不需要反复解释自己”。**

首轮POC用以下可测指标替代空泛满意度：

- 相关记忆召回准确率；
- 错误/无关记忆注入率；
- 新旧事实冲突处理正确率；
- 方案使用个人约束的比例；
- 行动创建率与按期跟进率；
- 用户修改/删除记忆后的传播正确率；
- 首字延迟、完整回复延迟和失败率；
- 越权读取、敏感日志和错误共享事件数。

### 2.3 与国内竞品的胜负边界

- 长期记忆：豆包已公开提供自动/显式记忆、查看和删除，记忆本身已是基线能力；
- 一次性通用问答：不承诺胜出；
- 生图、视频、PPT、办公功能广度：首版不竞争；
- 微信内低门槛进入：必须达到可用门槛；
- 连续使用后的个人理解：必须优于无记忆或弱记忆体验；
- 方案与行动持续性：必须成为核心差异；
- 记忆透明与用户控制：必须从首版存在。
- 回答级来源、有效时间、冲突演进、删除传播状态和行动结果回流：必须形成可验证的组合差异。

## 3. 用户体验与信息架构

### 3.1 双层入口与隐私模式

```text
微信客服：首次体验、随手问、普通轻对话、接收提醒
    │
    │ 私密话题提示＋一键续接，不要求重复描述
    ▼
小程序私密空间：深度对话、长期记忆、敏感内容、记忆与隐私控制
```

微信客服不是唯一对话空间，也不得伪装成人工客服：

- 首次提示明确“我是我在AI助理，不是人工客服”；
- 默认没有工作人员查看原始对话；人工排障只处理用户主动授权的指定内容；
- 首次提供“开始聊 / 进入私密空间 / 隐私怎么保护”三个动作；
- 普通工作和生活问题可留在客服通道；检测到健康、家庭、情绪、关系、身份或财务等潜在私密主题时，只提示切换，不擅自发送、共享或改变记忆状态；
- 一键进入小程序对应会话，服务端续接短期上下文，用户不需要重复描述；
- 小程序页面始终显示`私密对话 / 临时对话`状态，微信客服显示`便捷通道`，不使用含糊的“安全模式”文案；
- 两端只有在服务端完成可信身份绑定后才共享会话和记忆，不能仅凭前端传入的OpenID、手机号或会话ID合并账户。

微信客服首次提示控制在一屏内：

> 我是「我在」AI助理，不是人工客服。默认没有工作人员查看你的对话；只有你主动授权排查时，指定内容才可能由人工处理。你可以使用临时对话、查看或删除记忆。涉及私密话题，也可以切换到私密空间。

### 3.2 首版导航

首版只保留三个一级入口：

1. **对话**：默认首页，文字、按住说话、图片/文件后续渐进开放；
2. **行动**：AI建议转成的待办、跟进时间和结果；
3. **我的**：记忆中心、临时对话、数据与隐私、反馈。

“守望”“护你”“家庭关系”从首版主导航移除，后续作为能力模块出现。

### 3.3 对话页

- 微信式消息流；
- 文字输入和按住说话；
- 流式回复与明确停止按钮；
- 回复优先结论，长内容分层展开；
- 显示本次是否使用个人记忆；
- 可展开查看使用了哪些记忆及来源；
- 对回复提供“有帮助/没帮助/哪里不对”；
- 可将建议转成行动；
- 弱网、超时和模型降级有明确状态。
- 顶部明确显示当前入口和隐私模式，并允许切换临时对话；
- 从微信客服进入时显示“已续接上一段对话”，允许用户在发送前查看接续范围或选择新开会话。

### 3.4 记忆中心

分为：

- 已确认事实；
- 当前状态；
- 偏好与表达；
- 目标与进行中的事情；
- 时间线与阶段变化；
- 待确认候选；
- 已删除/撤回状态不向普通用户展示正文，只保留必要操作结果。

每条记忆显示：内容、类型、来源时间、确认方式、最近使用、可见范围和删除入口。

### 3.5 临时对话

- 不读取长期记忆；
- 不生成长期记忆候选；
- 只保留完成本轮会话所需的短期上下文；
- 页面必须明确标识当前为临时模式。
- 微信客服和小程序都能发起临时对话；跨入口跳转不得把临时会话升级为普通会话。

### 3.6 行动页

首版只做站内行动闭环：

- 标题、建议来源、计划时间、状态；
- 用户确认后才创建；
- 到期通过小程序允许的订阅消息或下次打开时跟进；
- 用户记录完成、未完成、效果和原因；
- 后续再接日历、提醒事项、外卖、出行或其他工具。

### 3.7 输入阶段顺序

1. 文字对话；
2. 按住说话＋ASR；
3. 图片理解；
4. 文件理解；
5. 老人简化模式；
6. 儿童年龄适配模式。

已有语音和图片UI可复用，但未接真实转写/理解前不能称为已支持。

## 4. AI对话、记忆与行动闭环

### 4.1 请求主链路

```text
小程序输入
→ Conversation API解析可信身份
→ 输入Schema、限流、幂等和安全检查
→ 意图识别与模式判断
→ Retriever检索相关记忆、时间线和行动结果
→ ContextBuilder构造最小必要上下文
→ Model Router选择模型
→ 流式生成与输出Schema校验
→ 返回回复、引用的记忆、待确认项和行动建议
→ 异步提取Observation / Memory Candidate / Action Candidate
```

### 4.2 记忆治理模型

```text
用户原始输入
→ Observation
→ 规则/模型提取Memory Candidate
→ 明确陈述按规则确认；推断保持候选
→ Profile Item
→ 混合检索
→ Context Injection
```

### 4.3 确认规则

- 用户明确陈述的事实可标记为`explicit_confirmed`，同时在记忆中心可见；
- 模型总结、关系判断、人格判断、风险判断只能是`candidate`；
- 敏感度只决定保护边界，不决定是否为真；
- 后续表达与旧条目不一致时，不直接覆盖，创建新版本并更新有效时间；
- 用户纠正立即生成新版本并使旧版本退出当前检索；
- 删除必须传播到Profile Item、索引、缓存和派生摘要。

### 4.4 记忆分类

- `stable_fact`：相对稳定的个人事实；
- `current_state`：阶段性状态；
- `preference`：表达、习惯和选择偏好；
- `goal`：当前目标；
- `decision_rule`：用户明确表达的判断原则；
- `relationship_boundary`：关系与分享边界；
- `action_result`：某次行动及结果；
- `candidate`：尚未确认的模型推断。

### 4.5 检索策略

首版使用可替换`Retriever`接口：

```text
候选召回：关键词＋Embedding＋时间/类型过滤
→ 去重和当前有效版本过滤
→ 相关性、时效性、置信度、敏感度排序
→ Token预算裁剪
→ 记录本轮实际注入项
```

POC数据量小，可在云函数中进行有限候选重排；正式规模前再决定腾讯云向量数据库、独立向量服务或其他实现，不把领域模型绑定到具体向量产品。

### 4.6 上下文分层

- 系统与安全规则；
- 当前用户明确设置；
- 本轮相关Profile Items；
- 相关时间线和行动结果；
- 最近对话摘要；
- 当前输入；
- 输出Schema。

禁止把完整聊天历史和完整画像无差别塞入Prompt。

### 4.7 输出结构

模型输出至少解析为：

```json
{
  "reply": "string",
  "memoryRefs": ["opaque-id"],
  "clarifications": [],
  "actionCandidates": [],
  "memoryCandidates": [],
  "safety": { "level": "normal", "reasonCode": null }
}
```

用户可见回复与后台结构化结果分开；解析失败时降级为纯回复，不把无效结构写入长期数据。

### 4.8 “越用越聪明”的实现边界

首版不对每位用户持续训练基础模型。产品能力随使用改善，来自四个可区分、可评测的层次：

1. 事实层：记住并正确更新用户事实、偏好、目标和阶段；
2. 理解层：根据纠正和反馈调整对约束、关系与决策规则的判断；
3. 程序层：把已验证的成功方案、失败原因和操作流程沉淀为可复用 playbook/skill；
4. 模型层：只有独立训练、评测和版本发布的参数更新才称为模型学习，不用上下文优化冒充模型变强。

POC-1验证事实层，POC-2强化理解层，POC-3在行动结果闭环上开始验证程序层。事实记忆和程序性技能都必须有来源、版本、适用范围、用户控制与回滚；一次偶然成功不能自动成为全局规则。

### 4.9 Hermes-lite智能体边界

基础模型决定通用推理上限；Hermes式“聪明感”主要来自按需加载Skill、相关记忆、工具调用、任务连续性和纠错复用。POC-1不只验证“模型能回复”，还必须验证一个最小受控Agent Loop：

```text
用户输入
→ Capability Router识别意图
→ Skill Registry加载最小必要规则
→ Retriever取得相关记忆
→ LLM决定直接回答或请求工具
→ Policy Engine校验权限、风险和步数
→ Tool Executor执行白名单工具
→ 工具结果返回LLM
→ 生成带来源的最终回复
→ Memory Observer只分析原始用户消息
```

POC-1只开放`general / personal_advice / factual_research`三类能力，Agent Loop最多3步。首批工具为`memory_search`、`current_time`和一个受控实时搜索适配器；读取类工具可自动执行，外发、写入、付费、删除、账号和其他不可逆操作必须显式确认。Skill首版由代码仓库版本化，只读加载，禁止模型自动创建或修改生产Skill。

                          │
                          ▼
Conversation API / Cloud Function
    │
    ├── Identity & Policy

    ├── Conversation Service
    ├── Agent Orchestrator
    │      ├── Capability Router
    │      ├── Skill Registry
    │      └── Tool Registry / Executor
    ├── ContextBuilder
    ├── Retriever ── MemoryStore
    ├── Model Router ── ModelAdapter ── CloudBase内置/自定义模型

    ├── Audit & Observability
    └── Async Jobs
             │
             ▼

客服通道和小程序可以使用同一记忆系统，但不是无条件共享：临时模式不读不写长期记忆；敏感记忆遵循确认与可见规则；渠道自身可能保留的数据不纳入“我在已彻底删除”的承诺范围。

## 6. CloudBase与模型策略

### 6.1 CloudBase职责

负责：

- 微信OpenID身份；
- 云函数/云托管；
- 文档数据库；
- 对象存储；
- 日志、指标和环境；
- 首版大模型统一接入。

不负责：

- 个人记忆的事实模型与时间线；
- Prompt和ContextBuilder；
- 记忆确认、冲突、删除和共享规则；
- 模型评测与产品路由；
- 行动跟进逻辑。

### 6.2 调用边界

小程序不直接调用模型。所有调用经自有服务端，保证：

- 密钥不下发；
- 记忆与权限一致；
- 可审计、限流和统计成本；
- 可切换模型和降级；
- 输出经过Schema和安全校验。

### 6.3 模型路由

首版建立三类路由，不先拍脑袋指定唯一模型：

| 路由 | 用途 | 目标 |
|---|---|---|
| fast | 普通聊天、分类、候选提取 | 低延迟、低成本 |
| reasoner | 复杂分析、方案、多约束决策 | 质量优先 |
| multimodal | 图片、文件、后续视频 | 理解准确、结构化输出稳定 |

候选来自CloudBase当前支持的DeepSeek、混元、MiniMax、Kimi、GLM；标准版以上可接兼容OpenAI/Anthropic协议的其他模型。未来可接通义千问或火山方舟，但产品逻辑不依赖单一厂商。

### 6.4 选型方法

- 使用纯虚构Persona和任务集；
- 同样的ContextBuilder和输出Schema；
- 测质量、延迟、结构化成功率、工具调用和成本；
- 至少测试两种模型后再指定默认路由；
- 模型更新必须重跑回归集；
- 后台保留模型、Prompt和Schema版本及回滚开关。

### 6.5 降级策略

- 强模型超时→快速模型；
- 结构化解析失败→重试一次或降级纯文本；
- 记忆检索失败→明确以无个人记忆模式回答；
- ASR失败→保留原录音并提示重试/文字输入；
- 全部模型不可用→保存本轮草稿，不假装已回答。

- `operation_jobs(status, nextRunAt)`；
- 幂等键唯一索引。

## 8. 权限、隐私与安全

### 8.1 角色

| 角色 | 可以 | 默认不可以 |
|---|---|---|
| 用户/数据所有者 | 对话、查看/改/删记忆、管理行动和导出 | 查看他人数据 |
| 被邀请亲友 | 使用自己的独立空间 | 自动读取邀请人的数据 |
| 协作者 | 查看明确授权的最小摘要或行动 | 查看原始对话和完整画像 |
| 运营只读 | 查看匿名指标、错误、版本和任务状态 | 查看原文、媒体和完整画像 |
| 隐私审计员 | 查看授权链和访问记录 | 修改业务数据 |
| 授权排障人员 | 在用户授权的片段、目的和期限内临时查看 | 搜索其他对话、扩大范围、长期保留正文 |
| AI任务执行器 | 按任务声明读取最小上下文 | 扩大范围、直接共享、绕过确认写记忆 |

### 8.2 硬规则

1. 邀请者、付款者、家属和管理员都不会自动成为数据所有者；
2. 身份只由服务端可信上下文取得；
3. 默认私密，分享必须明确、可预览、可撤回；
4. 首版运营后台不提供原文浏览入口；
5. 日志不写正文、媒体URL、完整OpenID、环境ID、Token和完整Prompt；
6. 测试只使用纯虚构Persona；
7. 高风险健康、法律、金钱和家庭敏感操作必须先确认；
8. 老人和儿童扩展前必须进行独立安全评审。
9. 健康、家庭、关系、身份、财务等敏感内容可以用于当前回答，但不得静默升级为稳定长期记忆；候选必须可见、可删，并按敏感级别要求确认。
10. 微信客服通道不得被描述成端到端私密空间；产品只能承诺自己控制的数据库、索引、缓存和派生数据，不承诺删除微信或腾讯依法、依约保留的通道数据。

### 8.3 人工排障授权

默认不存在运营人员浏览原始会话的入口。用户主动提交问题时，系统必须让用户预览并确认：

- 授权的具体消息或附件；
- 排障目的；
- 可访问人员或处理组；
- 到期时间；
- 是否允许生成脱敏复现材料。

授权产生不可篡改审计记录，到期自动撤销；查看、导出和再次转交都必须在授权范围内。普通客服工单只保存问题分类、错误码和处理状态，不自动附带原始会话。

### 8.4 删除流程

```text
用户删除
→ 立即阻断读取
→ 异步删除正文、媒体、Observation、Candidate、Profile Item、Embedding和缓存
→ 失败幂等重试
→ 审计只保留无内容的动作证明
→ 用户看到各步骤最终结果
```

删除结果必须分层展示：

- **我在可控范围：**消息正文、媒体、Observation、Candidate、Profile Item、Embedding、缓存、派生摘要与可定位备份任务；
- **必要无内容审计：**仅保留删除动作、时间、结果和不可逆散列，不保留正文；
- **外部通道范围：**微信客服或其他平台可能保留的数据按其规则处理，产品提供说明和可行的操作入口，不做无法验证的“全网彻底删除”承诺。

### 8.5 临时对话

临时对话不读取长期记忆，也不写Observation、Candidate、Profile Item和Embedding。技术日志仍只保留无内容的必要运行信息；渠道切换、重试和会话恢复都必须保留临时属性。

## 9. 最小后台与运维

### 9.1 首版不先建完整React后台

POC只实现支撑AI闭环的最小服务端能力：

- 匿名调用量、错误率、P50/P95延迟；
- 模型、Prompt和Schema版本；
- Token/资源点与成本上限；
- 异步任务失败和幂等重试；
- 反馈分类；
- 删除任务状态；
- 功能开关和紧急停用；
- 审计查询。

后台不提供按用户浏览聊天记录、全文搜索正文或批量导出媒体的能力。用户授权排障通过独立的`support grant`工作流实现，不复用运营只读权限。

首批个人AI闭环未通过前，不扩建用户管理、关系管理、内容管理和复杂运营工作台。

### 9.2 何时建设React管理Web

满足以下任一条件后进入：

- 有5名以上外部试点用户；
- 需要非开发人员处理反馈和删除请求；
- 模型路由、灰度和成本不能安全通过配置文件管理；
- 异步任务与审计需要可视化处理。

后台仍经`admin-api`访问，不直接连接数据库，不在浏览器持有管理密钥。

### 9.3 环境

| 环境 | 数据 | 用途 |
|---|---|---|
| local | InMemory Repository＋虚构数据 | 单元测试与领域开发 |
| test | 独立CloudBase环境＋虚构测试账号 | 集成、预览、模型和双账号测试 |
| production | 独立正式环境 | 小范围试点后启用 |

测试与生产不共享数据库、存储、管理员、环境ID和日志出口。

### 9.4 可观测性

- `requestId`贯穿输入、检索、模型、写入和异步任务；
- 指标：调用量、错误率、首字/P95延迟、模型降级、检索数量、结构解析失败、任务积压和成本；
- 告警不包含原文和完整标识；
- 模型请求日志只记录必要元数据和脱敏版本号。

## 10. 竞品与模型评测

### 10.1 竞品范围

首轮真实黑盒评测：

- 豆包；
- 腾讯元宝；
- Kimi；
- 千问；
- 我在 MyAlly 当前版本。

国际产品和开源项目用于能力参考，不作为中国用户可用性的直接替代：ChatGPT、Claude、Nomi、Kindroid、Second Me、Personal AI。

### 10.2 统一测试脚本

使用纯虚构Persona和30轮连续对话，测试：

1. 首轮通用回答；
2. 相关记忆召回；
3. 无关记忆抑制；
4. 新旧信息冲突；
5. 阶段变化；
6. 方案使用个人约束；
7. 不迎合与不确定性；
8. 行动创建与后续跟进；
9. 记忆来源、修改、删除和临时对话；
10. 语音入口摩擦、首字延迟和完整延迟；
11. 幻觉、敏感场景和越权共享。

### 10.3 三类结论分开

- **基础模型更强**：同样上下文下答案更好；
- **产品更懂用户**：记忆、检索和时间线让回答更贴合；
- **产品更好用**：入口、延迟、行动和控制更顺滑。

首版主要争取后两项，不用产品编排优势冒充基础模型领先。

### 10.4 开源审计

优先审计：

1. 自研 typed memory domain：先完成来源、确认、版本、有效时间和删除传播，作为POC-1基线；
2. Mem0与Graphiti：使用同一虚构中文评测集做离线对照，Graphiti重点验证双时态与来源追溯，Mem0重点验证提取和混合召回；
3. Letta、MemOS、Second Me：借鉴上下文层级、记忆类型和数据导入思想，不进入POC-1关键路径；
4. Agent Runtime：行动先用显式状态机和幂等任务，跨应用工具显著增加后再评估LangGraph/CloudBase Agent。

2026-07-17核验摘要与一手来源见`PRODUCT-HANDOFF.md`第5.7和6.1节。

复用前必须核验：LICENSE、NOTICE、维护状态、网络请求、数据外传、删除传播和依赖风险。只复用可替换模块，不整套照搬产品。

## 11. 分阶段交付路线

### POC-0｜旧代码与微信预览（已完成，非AI闭环）

已完成：

- 原生TypeScript小程序；
- 文字、语音、图片输入界面；
- `entries`记录、删除、分享状态领域规则；
- CloudBase云函数代码；
- 自动测试、TypeScript检查和微信预览；
- 本机AppID配置与Git隔离。

真实边界：测试号不能使用CloudBase，尚未完成真实云函数CRUD、模型、记忆、双账号和真机AI闭环。

### POC-1｜Hermes-lite成年人个人智能体最小闭环

**实施顺序：**先完成A段“小程序私密底座”，通过隐私验收后再开放B段“微信客服便捷入口”。客服接入不能阻塞小程序AI主链路，但也不能绕过隐私底座提前上线。

**A段范围｜小程序私密底座：**

- 正式小程序AppID和CloudBase测试环境；
- 对话首页替代记录首页；
- 文字输入＋流式AI回复；
- `Conversation API`、`ModelAdapter`和一个默认模型；
- `CapabilityRouter`、`SkillRegistry`、`ToolRegistry`、`AgentOrchestrator`和统一`PolicyEngine`；
- `general / personal_advice / factual_research`三类只读Skill；
- `memory_search / current_time / 受控实时搜索`三个白名单读取工具；
- 最多3步Agent Loop，工具结果真实返回模型后再形成最终回复；
- 消息存储、显式事实记忆、相关检索和第二轮注入；
- LLM Memory Observer只分析用户原始消息，结构化抽取失败不影响正常回答且不得伪造记忆成功；
- 记忆中心查看与删除；
- 临时对话；
- 回复反馈；
- 最小指标、审计、成本上限和降级。

**B段范围｜微信客服接入与无缝续聊：**

- 首次AI身份与隐私提示，以及“开始聊 / 进入私密空间 / 隐私怎么保护”三个动作；
- `ChannelAdapter`、可信身份绑定和内部`subjectId`；
- 客服便捷通道与小程序私密空间的清晰模式标识；
- 潜在私密话题只提示切换，不把敏感分类本身写成长期记忆；
- 短期、单次`handoffGrant`进入小程序对应会话，链接不携带正文；
- 跨入口续聊不重复描述，用户可在发送前查看接续范围或选择新开会话；
- 客服与小程序都支持临时模式，临时属性跨入口不降级；
- 无原文运营指标和用户授权排障工作流。

**退出条件：**

- 真机完成两轮对话，第二轮正确使用第一轮明确记忆；
- 真机问题至少一次由Capability Router选择正确Skill，并由模型真实调用一个白名单工具后使用工具结果回答；

## 15. Go / No-Go

### 进入首批成年人试点

- POC-1至POC-3有真实CloudBase和真机证据；
- 用户连续使用后记忆确实提高方案贴合度；
- 记忆查看、纠正、删除和临时对话可用；
- 行动跟进形成至少一个闭环；
- A/B隔离、日志脱敏和删除传播通过；
- 模型成本、延迟和失败有上限；
- 不需要开发者直接打开数据库处理日常问题。

### 开放微信客服便捷入口

- 小程序私密空间、临时对话、记忆中心和删除传播已先通过；
- 客服首条消息明确AI身份、人工访问例外与外部通道边界；
- 身份绑定、解绑、跨账号拒绝和handoff重放防护通过；
- 用户从客服切到小程序不必重复描述，并可查看接续范围或新开会话；
- 运营后台无原文入口，授权排障的范围、期限、撤销和审计可验证。

### 暂停扩展老人/儿童

- 成年人个人用户尚未持续自用；
- 记忆错误率高或删除不彻底；
- 仍依赖人工拼Prompt和手工改数据库；
- 高风险场景没有边界和升级机制；
- 邀请者可以默认读取被邀请者原始对话；
- 未完成对应年龄/无障碍真机测试。
